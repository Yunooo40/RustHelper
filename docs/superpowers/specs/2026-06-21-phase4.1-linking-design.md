# Phase 4.1 — Player linking (Discord ↔ Steam) — Design

Date : 2026-06-21 · Branche : `feat/phase4-linking` (depuis `main`, **sans** la 3.2 draft)

## Objectif
Lier un compte **Discord** à un compte **Steam/Rust**, par un flux à code qui prouve la
possession des deux côtés. C'est le **prérequis** des death notifications (4.2) et de tout
feature par-joueur.

## Flux (preuve de possession des 2 comptes)
1. Sur Discord : `/link` → génère un **code** court (6 car., TTL 10 min), réponse *ephemeral*
   « va en jeu et tape `!link <CODE>` ».
2. En jeu : `!link <CODE>` → le plugin POST `/link/claim { code, steam_id, steam_name }`
   (le serveur Rust atteste le SteamID = `IPlayer.Id`).
3. L'API matche le code → crée le lien `discord_user_id ↔ steam_id`, supprime le code.
4. `/unlink` supprime le lien.

## Schéma (db.js) — additif
```sql
link_codes ( code PK, discord_user_id, expires_at, created_at )      -- codes en attente
links ( id PK, discord_user_id UNIQUE, steam_id UNIQUE, steam_name, created_at, updated_at )
```
MVP : lien **1-to-1** (un Steam par Discord et inversement). Re-link = remplace l'ancien.

## Modèle `backend/models/link.js`
- `createCode({ discordUserId, ttlSeconds=600 })` → purge le code en attente de ce user,
  génère un code non ambigu (alphabet sans 0/O/1/I/L), insert, retourne `{ code, expiresAt }`.
- `claimCode({ code, steamId, steamName })` → `{ ok, reason?, link? }` :
  code inconnu → `unknown`, expiré → `expired` ; sinon supprime tout lien existant pour ce
  `discord_user_id` **ou** ce `steam_id` (garantit le 1-to-1), insert le lien, supprime le code.
- `findByDiscord`, `findBySteam`, `unlink(discordUserId)`, `purgeExpired()`.

## Routes `backend/routes/link.js`
- `POST /link/claim` (auth webhook, appelée par le plugin) :
  - champs manquants → **400** ; code inconnu → **404** ; expiré → **410** ; ok → **200**
    `{ ok:true, link:{ discord_user_id, steam_name } }`.
- `GET /link?discord=<id>` **ou** `?steam=<id>` → `{ ok:true, link|null }` (lecture).
- Montée dans `server.js` (`/link`) + ajout à la liste d'endpoints de `GET /`.
- `resetDb()` du harness de test étendu aux 2 nouvelles tables.

## Slash commands (thin, non unit-testées)
- `bot/commands/link.js` — `/link` : si déjà lié → montre le lien + suggère `/unlink` ;
  sinon `createCode` et explique le `!link <code>`. Réponse *ephemeral*.
- `bot/commands/unlink.js` — `/unlink`. Auto-chargées par `deploy-commands.js` + `bot.js`.

## Plugin `RustLinkRelay.cs` v0.3.0 (non unit-testable → fixture-contrat)
- `[Command("link")]` + `!link <code>` via `OnUserChat` → `DoLink(player, code)` :
  POST `ApiBase()/link/claim { code, steam_id=IPlayer.Id, steam_name=IPlayer.Name }`,
  réponse joueur selon le résultat (lié / code invalide / expiré / erreur).
- NB versions : la branche 3.2 (draft) est aussi en v0.3.0 ; au rebase de 3.2 sur le futur
  `main`, elle passera v0.4.0 (les 2 changements plugin se combinent trivialement).

## Tests (node:test)

- `tests/backend/link.test.js` :
