# Phase 7.2 — Auto-pairing FCM (auto-détection serveur) — Design

Date : 2026-06-24 · Branche : `claude/exciting-volta-l9dh9d` · Étend la Phase 7 (socle Rust+).

## Problème

La Phase 7 ouvre un socket Rust+ par serveur pairé, mais le pairing est **manuel** :
`/pair` exige que l'admin recopie 4 valeurs (`server_ip`, `app_port`, `steam_id`,
`player_token`) obtenues en lançant `npx @liamcottle/rustplus.js fcm-listen` sur sa machine
puis en cliquant « Pair » en jeu. Lourd, et à refaire à chaque serveur / rotation de token.

La vision : **un bot associé à un joueur Rust** détecte automatiquement le serveur où ce
joueur se pair, récupère infos/events temps réel et exécute des commandes pour son équipe.

## Réalité technique (lecture du code de `@liamcottle/rustplus.js` + `push-receiver`)

Facepunch livre les credentials via **notifications push FCM**. Quand le joueur clique « Pair
with Server » dans le menu Rust+ en jeu, une notif arrive avec `{ip, port, name, playerId,
playerToken, type:'server'}` (cf. `README.md` « Example Output » + `docs/PairingFlow.md`).

Le flux se décompose en **2 étapes de natures différentes** (`cli/index.js`) :

1. **`fcm-register`** (une fois) : `AndroidFCM.register` → token Expo → **login Steam via
   Google Chrome** (`linkSteamWithRustPlus` lance Chrome) → `registerWithRustPlus`. Produit
   `fcm_credentials.gcm.{androidId, securityToken}`. **Nécessite un navigateur + interaction
   humaine → NE PEUT PAS tourner dans le backend headless (Railway).** Reste **local**.
2. **`fcm-listen`** (permanent) : `new PushReceiverClient(androidId, securityToken, [])` +
   `.on('ON_DATA_RECEIVED'|'ON_NOTIFICATION_RECEIVED')` + `.connect()`. **Ça**, le backend
   peut le faire — c'est un socket longue-durée, comme nos connexions Rust+.

→ **Décision** : on garde `fcm-register` comme **étape locale documentée** (on ne réimplémente
pas Chrome+Steam). Le backend reçoit `{androidId, securityToken}` et fait tourner le **listener**.
Un jeu de credentials FCM = **un joueur Steam** (celui qui a fait l'OAuth) → mappe exactement
« le bot est associé à un joueur ».

## Décision — périmètre (borné)

Phase 7.2 = **réception des notifications de pairing *serveur* → création auto du pairing +
montée du socket live**. On NE traite PAS les notifs d'entités (smart switch/alarm, `type !==
'server'`) : ignorées proprement. On NE persiste PAS les `persistentId` entre redémarrages —
le ré-traitement d'une notif est **idempotent** (`findOrCreateByName`/`addServer` + `Pairings.add`
en upsert), donc rejouer quelques notifs récentes au boot est sans effet de bord.

## Constats (réutilisation)

- `backend/models/pairing.js::add()` upsert sur `UNIQUE(server_id, steam_id)` → idempotent.
- `backend/models/server.js` : `findOrCreateByName` (orpheline `guild_id NULL`) **et** `addServer`
  (adoption + 1er=défaut, garde anti-collision `SERVER_NAME_TAKEN`). On réutilise les deux.
- `rustplus/manager.js::syncServer(serverId)` : (re)monte le socket d'un serveur sans redémarrage.
  C'est l'unique point d'entrée pour passer live après ingestion des creds. **Injecté** dans le
  handler pour des tests sans socket.
- Pattern « secret jamais renvoyé / ephemeral / admin » : `routes/pair.js` + `commands/pair.js`.

## Modèle de données

Nouvelle table `fcm_credentials` (un listener par joueur enregistré) :

```
id, guild_id (TEXT null), discord_user_id (TEXT null), label (TEXT),
android_id (TEXT NOT NULL), security_token (TEXT NOT NULL  -- SECRET),
is_active (INTEGER DEFAULT 1), created_at, updated_at,
UNIQUE(android_id)
```

- `security_token` est un **SECRET** (reçoit les push de ce compte) → jamais loggé/renvoyé.
- `guild_id` (si enregistré via la commande Discord) → les serveurs auto-pairés atterrissent
  dans **ce** guild via `addServer`. Sinon (REST sans guild) → `findOrCreateByName` (orpheline).

## Comportement (handler de notification)

`handlePairingNotification(notification, credential, { syncServer })` :

1. `parsePairingNotification(notification)` → `null` si non-`server` / malformé → **ignore**.
2. Résout/crée le serveur : `credential.guild_id` présent → `addServer({guildId, name})`
   (capture `SERVER_NAME_TAKEN` → log + fallback `findOrCreateByName`) ; sinon
   `findOrCreateByName(name)`.
3. `Pairings.add({serverId, serverIp:ip, appPort:port, steamId:playerId, playerToken, label})`.
4. `syncServer(serverId)` → socket Rust+ live. Erreur de socket attrapée + loggée (jamais throw
   dans le callback du listener).

## Critères d'acceptation

1. `parsePairingNotification` : notif `type:'server'` → `{name, ip, port, playerId, playerToken}` ;
   `type:'entity'`/alarm → `null` ; body manquant/JSON invalide → `null` (jamais throw).
2. Handler : notif serveur (guild connu) → ligne `servers` (adoptée/créée, défaut OK) + ligne
   `rustplus_pairings` (token en DB, pas dans les logs) + `syncServer` appelé 1×. Rejouer la
   même notif → 0 doublon.
3. `POST /fcm {androidId, securityToken, label?}` (admin) → 200 + creds stockés (token jamais
   renvoyé) ; mauvais/zéro secret admin → 401 ; champ manquant → 400. `DELETE /fcm {androidId}`.
4. `/fcm` (Discord, admin, ephemeral) : `connect` ingère les 2 valeurs + démarre le listener ;
   `status` liste les listeners actifs ; `forget` retire. Token jamais ré-affiché.
5. `startFcmManager()` au boot ouvre 1 listener par credential actif ; no-op si
   `RUSTPLUS_ENABLED=false` ou `FCM_ENABLED=false` ou aucun credential. `stopFcmManager()` au shutdown.
6. Suite verte (étendue) + `npm run lint` clean ; `createBot` charge la commande ; `deploy-commands`
   sérialise `/fcm`.

## Changements de code (TDD)

- `config.js` : `rustplus.fcm.enabled` (`FCM_ENABLED !== 'false'`).
- `backend/db.js` : table `fcm_credentials` (+ index `is_active`).
- `backend/models/fcmCredential.js` : `add` (upsert `UNIQUE(android_id)`), `listActive`,
  `getByDiscordUser`, `getByAndroidId`, `remove`, `deactivate`.
- `rustplus/fcmParser.js` (**pur**) : `parsePairingNotification(notification)`. Cherche le body
  JSON (`notification.data.body` ou `appData`), parse, filtre `type==='server'`.
- `rustplus/fcmListener.js` : fin wrapper I/O de `PushReceiverClient` (start/stop, callback).
  Validé à la main au pairing (comme `connection.js` / le plugin) — pas de socket réel en test.
- `rustplus/fcmManager.js` : singleton. `startFcmManager()` (1 listener/credential actif),
  `syncCredential(id)` (après `/fcm connect`/`forget`), `stopFcmManager()`,
  `handlePairingNotification(notif, credential, {syncServer})` (cœur testable, `syncServer` injecté).
- `backend/routes/fcm.js` : `POST`/`DELETE /fcm` (admin) — monté dans `backend/server.js`.
- `bot/commands/fcm.js` : sous-commandes `connect`/`status`/`forget` (admin, ephemeral).
- `index.js` : `startFcmManager()` après `startManager()` ; `stopFcmManager()` dans `shutdown`.

  entité → null ; body absent/JSON cassé → null.
- `tests/rustplus/fcmManager.test.js` : `handlePairingNotification` avec DB `:memory:` + fake
  `syncServer` → serveur + pairing créés, `syncServer` appelé 1×, rejouer = idempotent ;
  notif entité → no-op. `startFcmManager` sans credential → no-op (pas de socket).

- `tests/backend/fcm-route.test.js` : 200/401/400 + DELETE + token jamais renvoyé.
- `tests/bot/commands.test.js` : `/fcm` présent + `toJSON()` OK.
