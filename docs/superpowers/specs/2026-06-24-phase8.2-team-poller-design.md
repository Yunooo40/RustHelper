# Phase 8.2 — Poller d'état d'équipe (connexions / morts / AFK) — Design

Date : 2026-06-24 · Branche : `feat/phase8.2-team-poller` · Suit la Phase 8.1 (commandes
in-game stateless).

## Problème

La 8.1 a livré des commandes *stateless* (1 appel Rust+ → 1 réponse). Il manque le **suivi
continu** de l'équipe : qui se connecte / se déconnecte, qui meurt, qui est AFK — sans que
personne ne demande. C'est un **mécanisme nouveau** (pas une entrée de plus dans le routeur) :
sonder `getTeamInfo` en boucle, **diffÉrencier deux snapshots**, et garder un **historique de
position** dans le temps pour l'AFK.

## Décision — périmètre

Phase 8.2 = **poller + annonces Discord + commande `!afk` + opt-in par serveur**. Quatre
catégories d'annonce, chacune **filtrée par la préférence du serveur** :

| kind            | déclencheur (diff entre 2 polls)        | défaut |
|-----------------|-----------------------------------------|--------|
| `join` / `leave`| `isOnline` change                       | **ON** |
| `death`         | `isAlive` true→false (membre online)    | **ON** |
| `afk` / `back`  | online + immobile ≥ seuil / re-bouge    | **OFF**|

Reporté (mécanismes distincts) :
- **8.3** : `!silence`/`!resume` (mute du bot in-game) + `!alarm`/`!remain`/`!stop` (timers
  planifiés, persistés, scheduler).
- Smart switches, alarmes Rust+, stockage, carte, distributeurs → **P10+**.

## Architecture

Nouveau module `rustplus/teamTracker.js` = **cœur pur** (testable sans socket) + une fine classe
`TeamTracker` que la `Connection` possède. Cohérent avec le découpage existant
(`connection.js` = transport, `router.js` = commandes, `teamFormat.js` = formatage pur).

- **`Connection`** : sur l'évènement `connected`, démarre `setInterval(poll, intervalMs)`
  (timer `.unref()`). `poll()` = `getTeamInfoAsync()` → `tracker.update(teamInfo, Date.now())`,
  erreurs loggées sans crash (comme le handler de chat). Sur `disconnected`/`stop` :
  `clearInterval` + `tracker.reset()`. Expose `this.tracker` (lu par `!afk`).
- **Baseline anti-spam** : le **1er `update` après (re)connexion enregistre l'état sans rien
  émettre** (`primed: false → true`). Sinon toute l'équipe serait annoncée « connectée » à
  chaque boot / reconnexion.
- **Kill-switch** : si `config.rustplus.enabled === false`, le manager est déjà idle → aucun
  poll. Aucun nouveau secret, aucune sortie réseau supplémentaire (juste des requêtes Rust+ sur
  le socket déjà ouvert).

## Cœur pur (`rustplus/teamTracker.js`)

Fonctions pures, sans I/O, testées directement :

- `diffMembers(prev, curr)` → `{ joined, left, died }` (listes de `{ steamId, name }`)
  - `joined` : `isOnline` dans `curr`, absent ou `!isOnline` dans `prev`.
  - `left` : `isOnline` dans `prev`, absent ou `!isOnline` dans `curr`.
  - `died` : membre online, `isAlive` passe `true → false` (gardé par `deathTime`). Une mort
    survenue pendant que le membre est offline est ignorée (pas de spam au retour).
- `computeAfk(posState, members, now, { thresholdMs, epsilon })` → `{ posState, nowAfk, returned }`
  - `posState` : `Map<steamId, { x, y, since }>`. Pour chaque membre **online** : s'il a bougé
    (`Math.hypot(x - px, y - py) > epsilon`) → `since = now` et position mise à jour ; sinon
    inchangé. Nouveau membre online → `since = now` (pas AFK d'emblée).
  - Est AFK = online && `now - since ≥ thresholdMs`. `nowAfk` = ceux qui **franchissent** le
    seuil ce tick (n'étaient pas comptés AFK avant). `returned` = étaient AFK et ont bougé.
  - Les membres offline / partis sont retirés du `posState`.

La classe `TeamTracker(serverId)` orchestre (état mutable minimal) :
- `update(teamInfo, now)` : si pas `primed` → mémorise `lastMembers`/`posState`, `primed = true`,
  **return** (aucune émission). Sinon : `diffMembers` + `computeAfk`, **émet `TEAM_EVENT`** par
  changement *filtré par les prefs du serveur*, puis met à jour `lastMembers`/`posState`.
- `getAfk(now)` → `[{ steamId, name, afkMs }]` des membres online actuellement AFK (pour `!afk`).
- `reset()` : `primed = false`, vide `lastMembers`/`posState` (sur déconnexion).

## Annonces (bus → Discord) — réutilise le pont existant

- `shared/bus.js` : `export const TEAM_EVENT = 'team-event';`
  Payload : `{ channelId, serverName, kind, member: { steamId, name }, afkMs? }`.
- À l'émission, le tracker résout `Servers.findById(serverId)` (→ `channel_id`, `name`) et lit
  `Servers.getNotifyPrefs(serverId)`. Émet **uniquement** les `kind` activés. Pas de
  `channel_id` (serveur pas encore `/setup`) → on n'émet pas.
- `bot/bot.js` : `bus.on(TEAM_EVENT, …)` → `sendToChannel(payload, teamEventEmbed(payload))`
  (le helper `sendToChannel` existe déjà et gère le `channelId` du payload).
- `bot/lib/embeds.js` : `teamEventEmbed(payload)` — couleur + emoji + libellé par `kind` :
  🟢 « Bob s'est connecté » · ⚫ « Bob s'est déconnecté » · 💀 « Bob est mort » ·
  💤 « Bob est AFK » · 🟢 « Bob est de retour ».

## Commande in-game `!afk` (`rustplus/router.js`)

- `!afk` → `sendTeamMessageAsync(formatAfk(client.tracker.getAfk(Date.now())))`.
- `formatAfk(list)` (pur, dans `teamFormat.js`) : trié par durée décroissante →
  `💤 AFK (2) : Bob (6m), Alice (12m)` ; liste vide → `✅ Personne d'AFK`.
- `!afk` est une **requête** : elle fonctionne même si l'annonce AFK est désactivée pour le
  serveur (l'opt-in ne filtre que les annonces Discord poussées, pas les réponses in-game).
- Nouvelle entrée dans la table `COMMANDS` du routeur ; le handler reçoit `(client, msg, args)`
  comme les autres (8.1).

## Opt-in par serveur (DB + `/notify`)

- Table `servers` (config par-serveur, déjà porteuse de `timezone`/`webhook_secret`) — 3 colonnes :
  `notify_connections INTEGER NOT NULL DEFAULT 1`, `notify_deaths INTEGER NOT NULL DEFAULT 1`,
  `notify_afk INTEGER NOT NULL DEFAULT 0`. Ajoutées au `CREATE TABLE servers` **et** via une
  migration idempotente `migrateNotifyPrefs(database)` : pour chaque colonne absente de
  `PRAGMA table_info('servers')`, `ALTER TABLE servers ADD COLUMN …`. Purement additif → pas de
  rebuild de table (contrairement à `migrateServers`). Exportée pour les tests, appelée après
  `migrateServers(db)`.
- `backend/models/server.js` :
  `getNotifyPrefs(serverId)` → `{ connections, deaths, afk }` (booléens) ;
  `setNotifyPref(serverId, key, on)` avec `key ∈ {connections, deaths, afk}` (mappe vers la
  colonne `notify_*`, rejette une clé inconnue).
- `bot/commands/notify.js` : slash command **admin** (même garde de permission que `/timer`,
  `/server-remove`). Options : `connections` / `deaths` / `afk` (booléens, optionnelles) +
  `server` (optionnel, résolu via `Servers.resolve(guildId, name)`). Sans option booléenne →
  **affiche** l'état courant ; avec → applique et confirme. Serveur introuvable → message clair.

## Config (`config.js`)

```js
rustplus.poll = {
  intervalMs:     Number(env.RUSTPLUS_POLL_MS ?? 20_000),   // 3 req/min, large sous les rate-limits
  afkThresholdMs: Number(env.RUSTPLUS_AFK_MS  ?? 300_000),  // 5 min immobile
  afkEpsilon:     Number(env.RUSTPLUS_AFK_EPSILON ?? 1.5),  // mètres : < => considéré immobile
}
```
Documenté dans `.env.example`.

## Tests (TDD, `node:test`, zéro dépendance)

- `tests/rustplus/teamTracker.test.js` :
  - `diffMembers` : join, leave, death (online true→false), absent→présent online, mort pendant
    offline ignorée, aucun changement → listes vides.
  - `computeAfk` : immobile franchit le seuil → `nowAfk` ; mouvement > epsilon reset `since` ;
    micro-mouvement < epsilon reste AFK ; AFK qui bouge → `returned` ; offline retiré du posState.
  - `TeamTracker` (DB `:memory:` seedée : 1 serveur + channel + prefs) : 1er `update` = baseline,
    **n'émet rien** ; 2e `update` avec un join → 1 `TEAM_EVENT` collecté sur le bus réel ; pref
    `afk = 0` → un `nowAfk` n'émet pas ; `getAfk` renvoie la bonne liste.
- `tests/rustplus/teamFormat.test.js` (étendu) : `formatAfk` (tri desc, formatage durée, vide).
- `tests/rustplus/router.test.js` (étendu, fake client) : `!afk` avec tracker seedé → liste
  formatée ; tracker vide → « Personne d'AFK ».
- `tests/backend/server.test.js` (étendu) : `getNotifyPrefs` défauts (1/1/0) ; `setNotifyPref`
  toggle + clé inconnue rejetée ; `migrateNotifyPrefs` idempotent (2 appels OK, valeurs
  préservées).
- Pattern bus : les tests s'abonnent à `TEAM_EVENT` et collectent, comme les tests webhook le
  font avec `RUST_EVENT`.

## Hors périmètre

- **8.3** : `!silence`/`!resume` + `!alarm`/`!remain`/`!stop` (état + persistance + scheduler).
- Smart switches, alarmes Rust+, monitors de stockage, carte → **P10+**.
- Persistance de l'état du poller : **en mémoire uniquement**. Au redémarrage on re-baseline
  (un cycle sans annonce) — acceptable, pas de table dédiée.
- Cooldown / anti-abus sur `!afk` : non (lecture seule, peu coûteux).
